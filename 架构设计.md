# AgentFlow 团队化重构迁移方案（独立新仓库）

## Summary

新建一个**独立仓库/独立目录**承接新的三层架构，再把现有能力按模块迁移过去。这样做的目标是：

- 避免旧架构与新架构在同一代码树里互相牵制
- 让三层资源模型、审批流、团队治理可以从一开始按正确边界设计
- 旧项目继续作为“功能参考实现”和“迁移来源”，直到新仓库稳定后再决定是否冻结或归档

新仓库建议定位为“团队版 AgentFlow Next”，保留 `agent-flow` 品牌与 CLI 入口，但内部结构按新模型重建。

## New Repository Shape

新仓库直接围绕三层资源模型组织，而不是从旧的 `cli/ + core/ + templates/` 继续长。

建议目录：

```text
agent-flow-next/
├── agent_flow/
│   ├── cli/
│   ├── core/
│   ├── resources/
│   ├── governance/
│   ├── runtime/
│   ├── adapters/
│   └── templates/
├── docs/
│   ├── architecture/
│   ├── migrations/
│   └── specs/
├── tests/
└── examples/
```

核心职责拆分：

- `core/`
  - 配置模型、层级绑定、路径规范、基础类型
- `resources/`
  - `skills/wiki/references/tools/souls` 的统一资产模型与解析器
- `governance/`
  - proposal、人工审核、AI 审核、finalize、审计记录
- `runtime/`
  - 运行时上下文注入、解析后资源视图、hook 装配
- `cli/`
  - `init / bind-team / asset / promote / doctor / sync`
- `templates/`
  - 全局默认 Soul、基础 hooks、初始化模版

## Architecture Decisions

### 1. 三层 AgentFlow 作为正式产品模型

#### 通用型 AgentFlow

位置：`~/.agent-flow/`

内容：

- 通用 `skills`
- 通用 `wiki`
- 通用 `references`
- 默认 `tools`
- 基础 hooks
- 不同角色 Agent 的默认 `Soul` 模版

#### 团队型 AgentFlow

位置：`~/.agent-flow/teams/{team_id}/`

内容：

- 团队 `skills/wiki/references/tools/hooks`
- 团队角色 `Soul.md`
- 团队治理配置
- 团队晋升 proposal 与审核记录
- Git 工作副本与同步状态

#### 项目级 AgentFlow

位置：`{project}/.agent-flow/`

内容：

- 项目专属 `skills/wiki/tools/hooks`
- 项目角色 `Soul.md`
- 项目绑定的 team 配置
- 项目运行态 `state`

注意：

- 项目层不再依赖旧版目录承载长期知识。
- 所有项目知识统一由 `.agent-flow/` 管理。

### 2. 统一资产模型

新仓库中将以下对象统一视为资产：

- `skill`
- `wiki`
- `reference`
- `tool`
- `hook`
- `soul`

统一元数据建议：

- `name`
- `kind`
- `layer`
- `owner`
- `version`
- `status`
- `review_state`
- `source_path`

统一解析优先级：

- `project > team > global`

例外：

- `references` 默认聚合
- `governance hooks` 不允许被项目层覆盖
- `souls` 按角色 fallback：`project -> team -> global`

### 3. 审核与晋升规则内建

#### 项目级 → 团队级

必须满足：

- 1 个团队负责人审核通过
- 1 个 AI 内容审核通过
- 全部通过后才能晋升

#### 团队级 → 通用型

必须满足：

- 至少 2 个团队负责人审核通过
- 至少 2 个不同 AI 审核 profile 通过
- 全部通过后才能晋升

审核载体固定为 Git/file-first：

```text
promotions/{proposal_id}/
├── proposal.yaml
├── snapshot/
├── human-reviews/
├── ai-reviews/
└── decision.yaml
```

AI 审核结果必须结构化落盘，至少包含：

- `profile`
- `decision`
- `summary`
- `risks`
- `reusability`
- `reasoning_digest`

### 4. CLI 按产品能力重组

新 CLI 建议以这几组命令为主：

- `agent-flow init`
  - `--global`
  - `--team`
  - `--project`

- `agent-flow bind-team <team_id>`

- `agent-flow asset`
  - `list`
  - `show`
  - `create`
  - `lint`
  - `resolve`

- `agent-flow promote`
  - `submit`
  - `review`
  - `ai-review`
  - `status`
  - `finalize`

- `agent-flow team`
  - `info`
  - `sync`
  - `members`
  - `policy`

- `agent-flow doctor`
  - 检查三层资源、绑定、同步、冲突、审核状态

旧仓库中的 `skill`、`team`、`doctor`、`init` 相关能力迁移时，优先映射到这些新命令，而不是原样照搬。

## Migration Strategy

### Phase 1: 建新仓库骨架

在新独立仓库中先建立这些最小能力：

- 基础 Python 包结构
- 新版 `init` 命令
- 三层目录初始化模版
- `ProjectConfig` / `TeamConfig` / `GlobalConfig`
- `AssetKind` / `ResourceLayer` / `ResolvedAsset`
- `ResourceResolver`
- 基础测试框架

此阶段不迁移旧实现逻辑，只建立新边界。

### Phase 2: 迁移“可复用基础能力”

从旧仓库迁移这些能力，但需要按新边界重写适配：

- 配置读写
- Team 绑定
- Team Git 同步
- Skill 管理的基础 CRUD
- Runtime context 的基础注入
- Doctor 的部分诊断逻辑

迁移原则：

- 迁移“能力”，不迁移旧目录假设
- 旧代码只能作为参考，不强行保留旧 API 形状
- 所有路径逻辑都改为走 `ResourceResolver`

### Phase 3: 落地统一资产体系

实现：

- `skills/wiki/references/tools/hooks/souls` 的统一扫描
- 同名覆盖与 shadow 诊断
- 角色 Soul fallback
- hook 分类：`runtime | governance`
- tool 元数据与来源层追踪

### Phase 4: 落地晋升治理

实现：

- proposal 提交
- 人工审核落盘
- AI 审核落盘
- finalize 校验
- 审计日志
- team repo 脏状态与未同步阻断

### Phase 5: 迁移旧知识资产

从旧仓库迁移知识类资源：

- `~/.agent-flow/skills/wiki/references` → 新 global assets
- `team` 相关目录 → 新 team assets
- 项目内 `.agent-flow` 中仍有价值的 `skills/wiki/soul/hooks/tools` → 新 project assets

迁移方式：

- 先写一次性 migration script
- 输出迁移报告
- 对无法自动分类的资源打 `manual-review-required`

### Phase 6: 切换与收尾

当新仓库达到可用标准后：

- 用真实项目验证初始化、绑定团队、资源解析、审核晋升
- 评估旧仓库去留：
  - 继续维护为 legacy
  - 只保留迁移文档
  - 归档只读

## Public Interfaces / Types

新仓库需要稳定这些接口：

- `GlobalConfig`
- `TeamConfig`
- `ProjectConfig`
- `AssetKind`
- `ResourceLayer`
- `AssetMetadata`
- `ResolvedAsset`
- `ResourceResolver`
- `PromotionProposal`
- `HumanReviewRecord`
- `AIReviewRecord`
- `PromotionDecision`

CLI 对外稳定入口：

- `init`
- `bind-team`
- `asset`
- `promote`
- `team`
- `doctor`

## Test Plan

必须覆盖：

- 新仓库初始化后可生成三层标准目录
- 项目绑定 team 后能正确解析 `project > team > global`
- 同名 skill/hook/soul 的覆盖与 fallback 正确
- governance hook 不能被项目层覆盖
- project → team 晋升缺任一审核时不能 finalize
- team → global 若不到 2 个负责人审核通过则不能 finalize
- team → global 若两个 AI 审核 profile 相同则不能 finalize
- 团队 repo 未同步或 dirty 时禁止 finalize
- 旧仓库导出的样例资产可被 migration script 正确导入
- doctor 能报告 shadow、缺失 soul、未完成 proposal、team sync 异常

## Assumptions

- 新架构从**全新独立仓库**开始，不在当前仓库中并行重构。
- 旧仓库是迁移来源，不是新架构的实现基底。
- 新仓库仍使用 Python CLI 技术栈。
- 所有功能由 `.agent-flow/` 统一承载，不再保留独立的开发工作流目录概念。
- Git 是团队层与审批流的事实源。
- v1 不做 Web UI，只做 CLI + 文件 + Git 工作流。
