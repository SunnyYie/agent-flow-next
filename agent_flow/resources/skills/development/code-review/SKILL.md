# Skill: code-review

> 灵感来源: agency-agents Code Reviewer + Archon 并行审查 — 四柱框架定义审查标准，并行编排定义多维度调度，两者互补不互斥。

## Trigger
当需要对代码变更进行审查、评审时触发。简单变更使用单 Agent 四柱审查；复杂变更启用并行审查流水线增强模式。

## Required Reading
- 铁律和验收标准（来自当前项目的 Agent.md 或 CLAUDE.md）
- 设计规格文档（用于交叉验证实现是否符合设计）

---

## Procedure

### Step 1: 了解变更范围 & 复杂度路由

1. `git diff` 查看变更内容，理解变更意图和上下文
2. 识别影响范围
3. **复杂度分类路由**，决定审查模式：

```
变更复杂度评估
├── 简单（文档/配置/样式/单文件微调）→ 单 Agent 四柱审查（仅 Step 2-4）
├── 中等（功能实现/Bug 修复/2-5 文件）→ 单 Agent 四柱审查 + 条件维度补充
└── 复杂（架构/安全/多模块/6+ 文件）→ 并行审查流水线（Step 2-7）
```

**路由原则**：不确定时启用更多维度，宁可过度审查也不遗漏。

---

### Step 2: 四柱审查框架（核心方法论）

所有审查模式的基础——每个审查维度内部都使用此框架。

#### Pillar 1: Correctness（正确性）

| 检查项 | 关注点 | 典型问题 |
|--------|--------|----------|
| 逻辑正确性 | 分支条件、边界值 | off-by-one、空列表未处理 |
| 类型一致性 | 参数类型、返回值类型 | 返回 dict 声称返回 TypedDict |
| 命名语义 | 变量/函数名是否表达意图 | `data` vs `workflow_state` |
| 实现完整性 | 是否覆盖所有指定功能 | 遗漏了设计文档中的字段 |

#### Pillar 2: Security（安全性）

| 检查项 | 关注点 | 典型问题 |
|--------|--------|----------|
| 输入校验 | 外部数据是否经过验证 | 未校验的用户输入 |
| 注入风险 | 命令/SQL拼接 | `f"rm {user_input}"` |
| 敏感数据 | 密钥/凭证暴露 | 硬编码 API Key |
| 权限边界 | Agent 操作范围 | 超出 Worktree 的文件写入 |

#### Pillar 3: Maintainability（可维护性）

| 检查项 | 关注点 | 典型问题 |
|--------|--------|----------|
| 代码重复 | 相同逻辑多处出现 | 复制粘贴的校验代码 |
| 函数长度 | 单函数是否过长 | >50行的函数 |
| 耦合度 | 模块间依赖是否合理 | 循环导入 |
| 文档适度 | 注释是否必要且准确 | 冗余注释、过时注释 |

#### Pillar 4: Performance（性能）

| 检查项 | 关注点 | 典型问题 |
|--------|--------|----------|
| 算法复杂度 | 是否有不必要的 O(n^2) | 嵌套循环查找 |
| 资源管理 | 文件/连接是否正确关闭 | 未使用 context manager |
| 异步误用 | async/await 是否必要 | 不需要异步的函数标记为 async |
| 内存占用 | 大对象是否及时释放 | 全局缓存无限增长 |

### Step 3: 严重性分级

#### CRITICAL — 必须立即修复，阻塞验收
- 安全漏洞（硬编码密钥、SQL注入、命令注入、XSS、认证绕过）
- 数据丢失或损坏风险
- 运行时必然崩溃的代码

#### HIGH — 应该修复，但不一定阻塞
- 逻辑错误（非预期行为）
- 遗漏的功能点
- 类型不一致
- 破坏API契约
- 关键路径缺少错误处理

#### MEDIUM — 建议修复
- 缺少输入校验
- 命名不清晰或逻辑混乱
- 重要行为缺少测试
- 性能问题（N+1查询、不必要的内存分配）
- 应提取的重复代码

#### LOW — 可选改进
- 风格不一致（linter未覆盖的）
- 命名改进建议
- 文档缺失
- 替代方案值得考虑
- 性能微优化

```markdown

{整体印象、主要关注点、亮点}

|---|------|------|------|------|

### HIGH (应该修复)
| # | 问题 | 位置 | 原因 | 建议 |
|---|------|------|------|------|

### MEDIUM (建议修复)
| # | 问题 | 位置 | 原因 | 建议 |
|---|------|------|------|------|

CRITICAL: {n}项 | HIGH: {n}项 | MEDIUM: {n}项 | LOW: {n}项 | 判定: {APPROVE/REQUEST_CHANGES}
```

### Step 5: 定义并行审查维度（仅复杂变更）

将审查拆分为独立的、可并行的审查维度。每个维度有：

| 要素 | 说明 |
|------|------|
| 明确的审查范围 | 不与其他维度重叠 |
| 独立的审查标准 | 可独立给出 PASS/FAIL |
| 结构化的输出格式 | 便于合并 |

**推荐维度拆分**：

| 维度 | 审查范围 | 触发条件 | 审查标准 |
|------|----------|----------|----------|
| 代码质量（必选） | 正确性、可维护性、性能 | 始终运行 | 四柱框架 Pillar 1/3/4 |
| 安全影响（条件） | 输入验证、权限边界、敏感数据 | 涉及认证/授权/数据处理 | 四柱框架 Pillar 2 |
| 错误处理（条件） | 边界情况、异常路径 | 涉及 I/O 或外部调用 | 安全检查 + 边界分析 |
| 测试覆盖（条件） | 覆盖率、测试质量 | 涉及逻辑变更 | testing_strategies.md |
| 文档一致性（条件） | 代码注释、文档与实现匹配 | 涉及 API 或配置变更 | 文档交叉验证 |

中等变更启用：代码质量 + 相关条件维度。复杂变更启用：全维度。

---

### Step 6: 并行执行与结果合并（Fan-Out / Fan-In）

#### Fan-Out — 并行执行

所有启用的审查维度同时启动：

```
                    ┌→ code-quality-agent   (fresh context)
                    ├→ security-agent       (fresh context)
implement ──────────├→ error-handling-agent (fresh context)
                    ├→ test-coverage-agent  (fresh context)
                    └→ docs-consistency-agent (fresh context)
```

**关键设计**：
- 每个审查维度使用 **fresh context**（新 AI 会话），确保审查独立性
- 每个维度各自独立访问代码库，输出结构化审查报告
- 并行维度之间不得有顺序依赖

#### Fan-In — 结果合并

收集所有维度的审查结果，合并为统一报告。

**合并规则**：

1. **按严重度排序**：CRITICAL → HIGH → MEDIUM → LOW
2. **去重**：不同维度发现的同一代码位置的问题只报告一次，标注来源维度
3. **冲突解决**：一个维度 PASS 另一个维度 FAIL → 以 FAIL 为准
4. **全量合并**：所有审查完成即合并，不管个别审查是否发现问题

```markdown
| # | 问题 | 位置 | 原因 | 建议 | 来源维度 |
```

### 修复循环

将审查报告传递给修复节点，按优先级处理：CRITICAL → HIGH → MEDIUM。

- CRITICAL 问题必须修复后重新验证
- 修复后可选择重新触发审查流水线（成本较高）或仅修复节点自检（成本较低）

## Rules

- 具体明确：指出具体行号和问题，不说"有安全问题"
- 解释原因：不只说改什么，还要解释为什么
- 建议而非命令："考虑用X因为Y" 而非 "改成X"
- 称赞好代码：指出巧妙的解决方案和整洁的模式
- 一次审查，完整反馈：不分散多轮评论
- 安全问题一律 CRITICAL，不降级
- 不提"看起来不错"类无依据评价——每个评价必须指向具体代码行
- 不建议超出任务范围的改动
- 修复建议必须具体可操作，不说"考虑重构"
- 各并行审查维度之间不得有顺序依赖
- 每个维度的审查输出必须是结构化格式
- 合并时同一代码位置的问题不重复报告，标注所有来源维度
- 复杂度分类必须保守——不确定时启用更多维度而非更少
- 并行审查的资源消耗与维度数成正比，需权衡成本与覆盖面
