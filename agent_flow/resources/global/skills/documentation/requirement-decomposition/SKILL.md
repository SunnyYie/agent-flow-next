# Skill: Requirement Decomposition & Code Mapping

> **需求拆解 → 逐项确认(含假设和边界) → 深度澄清 → 代码定位 → 设计决策确认 → 影响地图**。先确认"做什么"，再确认"怎么做"，最后才定位"改哪里"。**绝不假设理解了需求，每一点都必须确认，每个假设都必须穷举。**

## Trigger

当收到需求文档/需求描述时触发。完整流程：拆解 → 确认 → 澄清 → 定位 → 设计决策 → 映射。

## Required Reading

- `pre-flight-check` 技能（确保已执行知识检索）

## Procedure

### Phase 1: 需求通读与理解

1. **完整阅读**需求文档，不做任何假设
2. **识别需求类型**：新功能开发 / 功能增强 / Bug修复 / 重构
3. **理解业务上下文**：这个需求解决什么问题？为谁服务？

### Phase 2: 拆解为变更点

将需求拆解为**最小可独立的变更点**：

```markdown
# 需求拆解: {需求名称}

## 需求概述
{一句话描述需求目标}

## 变更点列表

### CP1: {变更点简述}
- 类型: {新增 | 修改 | 删除}
- 描述: {具体要做什么}
- 涉及端: {前端 | 后端 | 前后端}
- 我的理解: {用自己的话重述}
- 不确定项: {如果有模糊的地方，列出可能的解读}

## 变更点之间的依赖
- CP1 → CP2（CP2依赖CP1先完成）
- CP3 独立

## 需求范围边界
- 包含: {明确在范围内的}
- 不包含: {明确不在范围内的}
```

**拆解原则**：
- 一个功能点拆一个变更点，不要合并
- 埋点/日志类变更单独拆
- UI变更和逻辑变更分开
- 前端变更和后端变更分开

### Phase 3: 逐项确认（v3.0 增强：显性假设 + 边界场景）

**对每个变更点，必须向用户确认**：

```text
## 确认 CP{n}: {变更点简述}

我的理解：{用自己的话重述}
类型：{新增/修改/删除}
涉及端：{前端/后端/前后端}

### 显性假设
我做了以下假设：
1. {假设1} — 如果此假设错误，则{影响描述}
2. {假设2} — 如果此假设错误，则{影响描述}

### 边界场景
- 当{边界条件1}时，预期行为是{X}还是{Y}？
- 当{边界条件2}时，是否需要处理？

{如果有不确定项}：
❓ 不确定项：{具体问题}
  选项A: {解读1} — 优势{X}，劣势{Y}
  选项B: {解读2} — 优势{X}，劣势{Y}
  我的倾向：{A或B}，因为{理由}

请确认：以上理解和假设是否正确？边界场景如何处理？
```

**确认规则**：
1. 每个变更点**必须列出至少1个显性假设**，即使看起来很明显
2. 每个变更点**必须提出至少1个边界场景问题**，即使看起来不可能发生
3. 有不确定项 → 必须问，不能假设；必须列出各选项利弊
4. 没有不确定项 → 仍要确认假设和边界场景："CP{n}: {理解}，假设{X}，对吗？"
5. 用户纠正 → 更新描述，重新确认
6. 用户确认 → 标记为 ✅
7. 全部确认完 → 进入 Phase 3.5

**绝对禁止**：
- 跳过确认
- 合并多个变更点确认
- 自行决定模糊需求含义
- 只问"对吗？"而不列出假设和边界场景
- 假设列表为空（任何需求都至少有1个隐含假设）

### Phase 3.5: 深度澄清（v3.0 新增，不可跳过）

**触发条件**（满足任一即触发，几乎所有需求都会触发）：
1. Phase 3 中存在任何"不确定项"
2. Phase 3 中存在任何未用户确认的"显性假设"
3. 需求涉及跨模块修改（2+模块）
4. 需求类型为"新增"或"修改"（非纯删除）

**执行步骤**：

1. **假设穷举**：列出所有隐含假设（不仅是显性假设），分为4类：
   - 业务逻辑假设："用户一定是已登录状态"、"数据一定存在且格式正确"
   - 数据假设："接口一定返回非空"、"字段类型一定是X"
   - 环境假设："目标平台支持此API"、"旧版本客户端已升级"
   - 兼容性假设："此变更不影响其他功能"、"其他模块不依赖当前行为"

2. **反事实提问**：对每个关键假设提出反事实问题：
   - "如果用户未登录，这个功能怎么表现？"
   - "如果接口返回空数据，页面显示什么？"
   - "如果旧版本客户端没有升级，会怎样？"
   - "如果并发操作同一数据，会怎样？"

3. **影响追问**：对每个变更点追问影响范围：
   - "这个改动会影响哪些下游消费者？"
   - "有没有其他页面/组件也在用这个数据/接口？"
   - "这个修改是否会改变已有的行为/返回值？"
   - "删除/修改这个代码，哪些地方可能报错？"

4. **替代方案呈现**：对有2种以上解读的不确定项，呈现每种方案的利弊和影响范围

**输出**：将澄清结果更新到变更点描述中，标记已澄清项为 ✅

**产出标记**：澄清完成后，创建 `.agent-flow/state/.requirement-clarified`：
```
timestamp={ISO8601}
task={任务描述}
cp_count={变更点数量}
assumptions_resolved={已澄清假设数量}
uncertainties_resolved={已解决不确定项数量}
```

**绝对禁止**：带着未澄清的假设进入 Phase 4

### Phase 4: 输出确认文档

写入 `.agent-flow/state/requirement-decomposition.md`。

### Phase 5: 代码定位（分层精准搜索）

对每个已确认的变更点，提取3类搜索关键词：

```text
1. 业务实体词：功能名/模块名/页面名（如"实习圈"）
2. 技术特征词：技术概念（如"埋点"、"推送"、"OCR"）
3. 交互动作词：用户操作（如"发送消息"、"上传图片"）
```

**4层递进搜索**（由窄到宽，禁止跳层）：

```text
Layer 1: 精确匹配
  Grep "{业务实体词}" src/ --type-add 'web:*.{tsx,jsx,vue,svelte}' --type web
  Grep "{业务实体词}" src/ -t py -t ts

Layer 2: 模糊匹配（词根/变体）
  Grep "{词根}" src/
  Grep "{英文变体}" src/

Layer 3: 路由/配置匹配
  Grep "{路由关键词}" src/routes/ 或 src/pages/
  Grep "{配置关键词}" src/config/

Layer 4: 全局搜索（最后手段）
  Grep "{关键词}" .  (排除 node_modules/ .git/ dist/)
```

**绝对禁止**：跳过 Layer 1-3 直接做全项目扫描。保留原文用词，中文用中文搜索。

**逐点确认映射**：
```text
CP{n}: {变更点简述}
定位结果：需要修改 {文件路径}:{行号或函数名}
修改原因：{为什么要改这个文件}
搜索依据：{关键词} → Layer {n} → {匹配说明}
对吗？有没有遗漏或错误？
```

用户可能直接告诉你要改哪里——优先采纳用户指定的位置。

### Phase 5.5: 设计决策检查点（v3.0 新增，不可跳过）

**目的**：在定位代码后、编写影响地图前，与用户确认关键设计决策。**这是防止"定位对了但改法错了"的关键环节。**

**必须确认的设计决策**（对每个变更点）：

1. **修改方式确认**：
   - CP{n}: 是{新增函数/修改已有函数/重构模块/新增文件}？
   - 为什么选择这种方式而不是其他方式？
   - 替代方案是什么？未选原因？

2. **影响范围确认**：
   - 直接影响：{列出需要修改的文件}
   - 间接影响：{列出可能受影响的调用方/消费者}
   - 不受影响：{明确说明哪些看起来相关但实际不需要改动}

3. **实施策略确认**：
   - 是一次完成还是分步实施？
   - 是否需要临时兼容方案（如旧接口保留N个版本）？
   - 是否需要 feature flag 控制？

4. **回滚方案**：
   - 如果出了问题，如何快速回滚？
   - 是否需要灰度发布？

**输出模板**：
```text
## 设计决策: CP{n}

### 修改方式
{描述}，原因：{理由}
替代方案：{其他方式}，未选原因：{理由}

### 影响范围
- 直接修改: {文件列表}
- 可能影响: {文件列表}（需要测试验证）
- 确认无关: {文件列表}（原因：{说明}）

### 实施策略
{描述}

### 回滚方案
{描述}

确认：以上设计决策是否可以？有没有需要调整的？
```

**产出标记**：设计决策确认后，创建 `.agent-flow/state/.design-confirmed`：
```
timestamp={ISO8601}
task={任务描述}
design_decisions={变更点数量}
rollback_plans={有回滚方案的变更点数量}
```

**绝对禁止**：
- 未经用户确认设计决策就生成代码影响地图
- 合并多个变更点的设计决策一起确认
- 自行决定有争议的设计方案
- 设计决策中没有替代方案和回滚方案

### Phase 6: 生成代码影响地图（v3.0 增强）

写入 `.agent-flow/state/code-impact-map.md`：

```markdown
# 代码影响地图: {需求名称}

| 变更点 | 类型 | 文件路径 | 修改内容 | 实施策略 | 回滚方案 | 端侧 | 确认 |
|--------|------|---------|---------|---------|---------|------|------|
| CP1: xxx | 新增 | src/xxx.tsx:12 | 具体修改 | 一次完成 | 删除新文件 | 前端 | ✅ |

## 明确无关的代码
- src/yyy/ — 原因

## 实施顺序建议
1. CP{n}（原因）
```

## Rules

- **不假设理解**：每个变更点都必须与用户确认，每个假设都必须显式列出
- **逐项确认**：不能合并确认，必须逐个
- **显性假设**：每个变更点至少1个假设，假设不能为空
- **边界场景**：每个变更点至少1个边界问题，不能只问"对吗？"
- **深度澄清**：Phase 3.5 不可跳过，假设穷举、反事实提问、影响追问缺一不可
- **设计决策**：Phase 5.5 不可跳过，修改方式、影响范围、实施策略、回滚方案缺一不可
- **精准搜索优先**：永远从 Layer 1 开始，不要直接全项目扫描
- **保留原文用词**：中文关键词用中文搜索，不要翻译成英文再搜
- **映射也要确认**：代码定位结果同样需要逐项确认
- **先确认做什么，再确认怎么做，最后才找改哪里**：Phase 3 → 3.5 → 5 → 5.5 的顺序不可打乱
- **标记驱动**：Phase 3.5 和 5.5 完成后必须创建标记文件，hook 会检查
